📌 세션 기반 인증
동작 흐름

    1.유저 로그인 → 서버 메모리에 세션 생성
    2.세션을 구분하는 Session ID 발급
    3.브라우저 쿠키에 Session ID 저장
    4.브라우저는 모든 요청에 Session ID 자동 포함
    5.서버가 Session ID 비교 → 일치하면 인증/인가 처리

장점
    서버가 직접 세션 관리해서 구현 쉽고 안정적
    로그인 상태 확인/관리하기 쉬움

단점
    유저 많아지면 서버 메모리/DB 부담 증가
    멀티 디바이스 / 중복 로그인 관리 문제 생김
    서버 여러 대 쓰는 로드 밸런싱 환경에서 세션 공유가 복잡함

        로드 밸런싱
        요청을 여러 서버에 나누는 방식
        문제: 각 서버가 서로 동일한 세션 상태를 알아야 해서 관리가 어려움



📌 토큰 기반 인증(JWT)
동작 흐름

    1.유저 로그인 성공 → 서버가 Secret Key 로 JWT 발급
    2.클라이언트가 JWT 저장(브라우저, 앱 등)
    3.요청할 때마다 JWT를 함께 전송
    4.서버는 DB 조회 없이 JWT 자체를 검증해서 인증 처리

JWT 구조
    Header
    어떤 방식으로 서명했는지(alg), 타입은 JWT인지 등 메타 정보

    Payload
    유저 정보, 권한, 만료 시간 등 들어 있음
    Base64 인코딩이라 누구나 디코딩 가능 → 민감 정보 넣으면 안 됨

    Signature
    Header + Payload 를 Secret Key 로 암호화 시킨 값
    이걸로 토큰 위조 여부 판단

장점
    서버에 유저 세션 저장 안 해서 메모리 부담 없음
    서버 여러 대(로드밸런싱)에서도 추가 관리 필요 없음
    여러 디바이스에서도 문제없음

단점
    Payload가 인코딩일 뿐이라 내용이 노출될 수 있음
    Secret Key 유출되면 위험
    보안 위해 만료시간을 짧게 설정해야 함(몇 분~1시간 정도)
    중요 정보는 Payload에 넣지 말기 → 필요하면 JWE(암호화된 JWT) 사용

Base64 간단 정리
    데이터를 문자 깨짐 없이 전달하기 위한 인코딩 방식
    암호화가 아니다 → 누구나 디코딩 가능
